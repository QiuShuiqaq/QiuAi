# QiuAi 结果导出后自动清理设计

## 背景

QiuAi 当前“结果导出”流程在批量下载成功后，会保留原始结果文件夹。随着客户长期使用，`DATA/output` 下的结果目录会持续累积，导致：

- 结果导出列表越来越长
- 本地存储占用不断增加
- 用户需要手动逐个删除历史结果
- 非技术用户容易遗忘清理

用户提出的新需求是：在“批量下载”后，支持自动删除已经成功导出的结果文件夹，以保持导出区和输出目录整洁。

## 目标

在不改变现有“批量下载”为 zip 导出的前提下，为“结果导出”模块增加一个**长期偏好开关**：

- 开关位置：结果导出面板内
- 作用范围：工作室四个设计模块的结果导出
- 存储方式：本地 settings
- 默认值：开启

当开关开启时：

- 仅在 zip 导出成功后
- 自动删除本次勾选下载的结果文件夹
- 自动刷新结果导出列表
- 清空当前已选择项

## 非目标

本次改动不做以下事情：

- 不改变导出格式，仍然使用 zip 包导出
- 不增加“下载到哪里”的新逻辑
- 不把自动清理做成临时勾选
- 不改动工作台任务队列逻辑
- 不清理未被选中的历史结果
- 不清理失败导出的结果

## 用户体验设计

### 1. 开关位置

开关放在 `ResultExportPanel.vue` 的底部操作区，靠近“批量下载”按钮。

推荐文案：

`下载后自动删除已导出的结果文件夹`

这是一个长期偏好，不是一次性选项。

### 2. 默认状态

默认值为：`开启`

原因：

- 用户已明确提出希望减少历史结果堆积
- 当前产品面向普通用户，应尽量减少后续手工维护

### 3. 行为规则

#### 开启时

当用户点击“批量下载”并且 zip 成功导出后：

1. 删除本次勾选的结果文件夹
2. 刷新结果导出列表
3. 清空已选择项
4. 给出成功提示

#### 关闭时

保持现有行为：

1. 导出 zip
2. 保留原始结果文件夹
3. 不做额外删除

### 4. 失败处理

#### 导出失败

- 不删除任何结果目录
- 维持当前错误提示逻辑

#### 导出成功、删除失败

- zip 导出仍然视为成功
- 需要额外提示“下载成功，但部分结果文件夹删除失败”
- 刷新结果列表，以真实文件系统状态为准

#### 用户取消保存

- 不导出
- 不删除

## 数据设计

### Settings 新增字段

在 `settingsStoreService` 中新增布尔配置字段：

- `downloadCleanupEnabled: true`

要求：

- 纳入默认配置
- 纳入 normalizeSettings
- 通过现有 settings IPC 存取

### 渲染层状态

`App.vue` 读取并维护该设置，用于：

- 传递给 `ResultExportPanel`
- 在用户切换时即时保存到 settings

## 前端设计

### ResultExportPanel

新增 props：

- `downloadCleanupEnabled`

新增 emit：

- `toggle-download-cleanup`

展示形式：

- 一个 checkbox / switch
- 配一个简短说明文案

### App / DesignWorkspace / TaskManagerSidebar

需要把该状态从 `App.vue` 逐层透传给：

- `DesignWorkspace.vue`
- `TaskManagerSidebar.vue`
- `ResultExportPanel.vue`

并提供切换事件上抛回 `App.vue`。

## 后端设计

### 现有导出链路

当前导出链路为：

1. renderer 调用 `exportStudioResults`
2. `studioIpc` 弹出保存对话框
3. `studioWorkspaceService.exportSelectedResults()` 生成 zip

### 自动清理策略

本次不把“导出 + 删除”强耦合进 `studioWorkspaceService.exportSelectedResults()`

原因：

- 当前 `exportSelectedResults()` 的职责很单一，只负责导出
- 删除逻辑已有 `deleteExportItem()`
- 保持导出与删除解耦，便于失败提示和逐项回收

因此推荐：

1. 先调用 `exportStudioResults`
2. 若返回成功且 `downloadCleanupEnabled === true`
3. 由 renderer 逐个调用 `deleteStudioExportItem`
4. 删除完成后刷新 studio snapshot

这样可复用现有删除逻辑，并减少主进程接口改动。

## 消息反馈设计

### 下载成功且删除成功

提示：

`批量下载成功，已自动清理已导出的结果文件夹`

### 下载成功但删除部分失败

提示：

`批量下载成功，但部分结果文件夹自动清理失败`

### 下载成功且开关关闭

保持现有成功提示即可。

## 测试设计

### 渲染层

需要补充源码测试断言：

- `ResultExportPanel` 包含自动清理开关文案
- `App.vue` 包含该设置状态与切换逻辑
- 透传链路存在

### 后端 / 集成逻辑

重点验证：

1. 开关默认值为 true
2. 下载成功且开关开启时，会触发删除逻辑
3. 下载成功但开关关闭时，不触发删除逻辑
4. 下载失败时，不触发删除逻辑
5. 删除部分失败时，仍保留导出成功结果反馈

## 风险与权衡

### 风险

- 用户可能没意识到下载后原始文件夹被清掉

### 应对

- 用明确的开关文案说明行为
- 默认开启，但允许用户主动关闭

### 工程权衡

采用 renderer 串联“导出成功后逐个删除”的方案，而不是新建一个“导出后自动清理”的主进程复合接口，是因为：

- 现有架构已具备单项导出与单项删除能力
- 改动更小
- 测试边界更清晰
- 出错时更容易拆分反馈

## 实施结论

本次改动采用：

- 结果导出页内长期偏好开关
- 默认开启
- 导出成功后按勾选项自动清理
- 清理失败不回滚下载成功结果
- 设置持久化到本地
