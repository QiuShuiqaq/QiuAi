# QiuAi API-Key 用户锁定与管理员隐藏维护设计

## 背景

QiuAi 当前工作台右侧存在 `全局 API-Key 配置` 区域，普通用户可以直接：

- 查看 API-Key 输入框
- 编辑 API-Key
- 保存 API-Key
- 切换启用中的 Key 槽位

这不符合当前产品的实际运营方式。根据现有使用场景：

- 软件面向普通用户
- 测试和交付阶段由服务方代为配置 API-Key
- 用户不应接触或修改 API-Key
- 你本人仍需要保留一个可维护入口，方便远程协助客户时更新 Key

因此，需要把 API-Key 管理改造成“普通用户完全不可见、不可改；服务方可通过隐藏管理员入口维护”的模式。

## 目标

在不影响现有图片生成和本地工作流的前提下，实现以下能力：

- 普通用户在前端界面看不到 API-Key 内容
- 普通用户不能切换 API-Key
- 普通用户不能通过普通设置保存链路修改 API-Key
- 通过隐藏入口触发管理员验证
- 管理员输入固定密码 `qiuai@123` 后，才能打开临时 API-Key 配置面板
- 管理员面板中仅保留单个 API-Key 输入与保存能力
- 管理员状态只在当前运行期有效，程序关闭后失效

## 非目标

本次改动不做以下内容：

- 不引入网站账号体系
- 不做服务端用户权限管理
- 不做多级管理员角色
- 不做复杂加密方案
- 不保留双 Key 可视化切换交互
- 不在普通工作台直接暴露 Key 掩码或 Key 状态明细

## 用户体验设计

### 1. 普通用户界面

工作台原有 `全局 API-Key 配置` 区域改造成服务配置说明区。

普通用户仅看到静态说明，例如：

- `服务配置状态`
- `API 已由服务方预配置`
- `如需更换配置，请联系服务方`

不显示以下内容：

- `API-Key 1`
- `API-Key 2`
- 输入框
- 切换按钮
- 保存按钮
- 任何 Key 掩码内容

### 2. 隐藏管理员入口

顶部 Logo 区作为隐藏触发点。

建议交互：

- 连续点击 Logo 5 次
- 触发管理员验证弹窗

此入口不做界面提示，避免普通用户误触后理解为常规功能。

### 3. 管理员验证弹窗

隐藏入口触发后，先弹出管理员验证框，而不是直接展示配置面板。

弹窗内容：

- 密码输入框
- `确认` 按钮
- `取消` 按钮

固定密码：

- `qiuai@123`

行为规则：

- 输入正确：进入临时管理员模式，并打开管理员配置面板
- 输入错误：提示 `管理员验证失败：密码错误`
- 点击取消：关闭弹窗，不进入管理员模式

### 4. 管理员配置面板

管理员验证通过后，再显示一个独立的小面板或弹窗。

仅保留：

- 单个 `API-Key` 输入框
- `保存` 按钮

不保留：

- 双槽位展示
- 启用槽位切换
- 普通保存配置入口

保存成功后：

- 更新本地配置
- 给出成功消息反馈
- 管理员面板可继续保留打开，方便重复调整；也可以保存后关闭，本次建议保留打开，由用户自行关闭

## 权限与状态设计

### 1. 普通用户态

默认启动状态即普通用户态。

普通用户态下：

- 前端无 Key 可见性
- 前端无 Key 编辑能力
- 普通 settings 保存接口不可改 Key

### 2. 管理员态

管理员态是一个仅存在于当前运行期内的临时状态。

特点：

- 只有密码验证通过才进入
- 不持久化到 settings
- 程序关闭后自动失效

建议维护一个渲染层状态：

- `isAdminApiConfigUnlocked`

仅用于控制管理员面板显示与管理员保存入口的可用性。

## 数据与接口设计

### 1. 普通设置保存接口

现有普通设置保存链路继续保留给以下配置使用：

- 上传目录
- 主题
- 积分
- 下载后自动清理偏好
- 其他非敏感本地设置

但必须禁止通过该链路修改以下字段：

- `apiKeys`
- `apiKey`
- `activeApiKeyIndex`

如果普通请求试图修改这些字段，后端直接拒绝并返回错误：

- `当前版本不允许用户修改 API-Key`

### 2. 管理员专用保存接口

新增单独 IPC，例如：

- `settings:save-admin-api-key`

职责单一：

- 保存一个新的 API-Key

前端提交结构可简化为：

- `apiKey`
- `password`

后端验证：

1. 校验密码是否为 `qiuai@123`
2. 校验 Key 是否为非空字符串
3. 保存到现有 settings 结构

### 3. 与当前双槽位结构的兼容

当前项目底层存储是：

- `apiKeys`
- `activeApiKeyIndex`

为了最小改动兼容旧逻辑，本次建议：

- 管理员保存单 Key 时，始终写入 `apiKeys[0]`
- 同时把 `activeApiKeyIndex` 固定写为 `0`
- `apiKey` 同步为该值

这样：

- 后端生成链路无需大改
- 前端用户侧彻底不需要感知双 Key 结构
- 未来若需要恢复多槽位，可内部兼容扩展

## 前端设计

### 1. 工作台面板

`WorkspaceDashboard.vue` 中原有 `全局 API-Key 配置` 区域改为静态服务说明区。

不再接收普通 Key 编辑事件。

### 2. 顶部栏

`AppTopBar.vue` 的 Logo 点击事件保留，并在 `App.vue` 中增加隐藏点击计数逻辑：

- 连续点击计数
- 达到阈值后弹出管理员密码框
- 成功后重置计数

### 3. 管理员弹窗

管理员验证与管理员配置建议做成独立弹窗组件，避免把逻辑继续堆积在工作台内。

推荐拆成两个组件：

- `AdminPasswordDialog.vue`
- `AdminApiKeyDialog.vue`

如果希望减少文件数，也可以合并成一个二阶段弹窗组件，但推荐拆开以保持职责清晰。

## 后端设计

### 1. settingsStoreService

在 `saveSettings()` 中加入敏感字段拦截逻辑。

普通调用只要包含任一敏感字段：

- `apiKeys`
- `apiKey`
- `activeApiKeyIndex`

则直接抛错。

### 2. 新管理员服务入口

新增一个管理员保存函数，例如：

- `saveAdminApiKey({ apiKey, password })`

逻辑：

1. 校验密码
2. 读取当前设置
3. 把 `apiKeys[0]` 写成新值
4. 把 `activeApiKeyIndex` 固定成 `0`
5. 返回更新后的安全结果

### 3. 安全回传

管理员保存成功后，回传给渲染层的数据不必返回完整 `apiKeys` 数组用于普通展示。

更适合回传：

- 保存成功状态
- 更新时间

但考虑当前 `App.vue` 可能仍依赖设置结果，最小实现时也可以回传完整 settings，只要普通界面不展示即可。

## 错误处理

### 普通用户越权修改

返回：

- `当前版本不允许用户修改 API-Key`

### 管理员密码错误

返回：

- `管理员验证失败：密码错误`

### 空 Key 保存

返回：

- `API-Key 不能为空`

### 其他保存异常

返回：

- `管理员保存 API-Key 失败：具体错误`

## 测试设计

### 后端

需要新增测试覆盖：

1. 普通 `saveSettings()` 不能修改 Key
2. 管理员密码错误会失败
3. 管理员密码正确时可保存 Key
4. 保存后 `apiKeys[0]` 被更新，`activeApiKeyIndex` 为 `0`

### 前端源码测试

需要补充断言：

1. 工作台不再包含普通 API-Key 输入框与切换按钮
2. 存在管理员密码弹窗与管理员配置弹窗
3. 顶部 Logo 点击会触发隐藏管理员验证链路

## 风险与权衡

### 风险

- 固定密码写在本地程序里，不属于高强度安全方案

### 可接受原因

- 当前产品定位为本地桌面软件
- 面向非技术用户
- 你的核心诉求是“普通用户改不了”，不是抵抗逆向分析

### 工程权衡

相比引入复杂账号系统或外部授权中心，本方案：

- 改动范围小
- 对现有本地工作流影响最小
- 足以满足当前交付阶段的控制需求

## 实施结论

本次改动采用以下方案：

- 普通用户完全看不到 API-Key 内容
- 普通用户不能切换、不能修改 API-Key
- 普通 settings 保存链路后端硬拦截 Key 修改
- 通过顶部 Logo 连点 5 次触发管理员密码验证
- 使用固定密码 `qiuai@123`
- 验证通过后打开单 Key 管理员配置面板
- 管理员态只在当前运行期有效
